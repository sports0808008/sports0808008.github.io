day071Redis

1.Redis的概念

​ 思考一个问题：我们访问mysql的数据库的时候是通过什么方式去访问的呢？

​ 回答：通过建立连接使用io（input和output）的形式去访问磁盘，访问磁盘的速度是不快的，这种情况下就不满足高并发的需求，这种背景下一种叫做nosql的数据库就产生了，这种数据库的数据保存在内存中，保存到内存中的数据一旦系统重启就会丢失，所以很多nosql的数据就提供了持久化的支持（将内存中的数据同步到到磁盘中），这样从内存中读取数据速度就非常快

​ 补充：结构化数据就需要使用sql语句进行操作。

​ Redis是一个NoSql数据库，不使用sql语句实现数据的CRUD，而且数据是保存到内存中的，那么在内存中的数据操作是非常迅速的，而我们之前的mysql等关系型数据库的数据是保存到磁盘中的，访问的速度是相对较慢的，不适合于高并发的访问。而Redis适合于高并发的访问，可以实现每秒10W次的访问量（需要你的电脑的硬件好，如果你的电脑性能一般的话可以实实现到2w左右），在Redis之前出现过一个memcached数据库，也是一款NoSql数据库，两者之间有什么区别呢？

​ 面试题：Redis和memcached的区别是什么？

​ 1.Redis的数据不但可以保存到内存中还可以持久化到磁盘中，而memcahed只能保存到内存中，不能持久化到磁盘。

​ 2.Redis可以保存key—value结构的数据,其中value的类型可以是list、set、hash等结构的数据，而memcached保存key—value结构的数据，但是value只有一种类型，就是字符串。

       3.Redis是单线程模型（这里指的是核心操作模块是一个线程独立完成的，除了操作核心模块的线程之外还有一些辅助性的线程存在，而且在redis6版本的时候就**部分核心操作也交由了多个线程去完成**），而memcached是多线程模型

​ Redis是现在企业要求必须会使用的，也是面试必须要掌握的技术，当然也是工作中你必须要掌握的技能。

8.7.1 使用 Set 实现

一个用户一天内多次访问一个网站只能算作一次，所以很容易就想到通过 Redis 的 Set 集合来实现

​ set 类型数据保存了是不会重复的，现在是能实现需求了，但是存在一个问题，那就是如果有千万或者上亿级别的访问量，就会导致这个 set 集合保存的数据太多，出现内存耗费问题，于是 HyperLogLog 就登场了

8.7.2 什么是 Hyperloglog?

​ 个在大数据量下统计基数的算法，占用内存小，误差小，但是会损失一定精度(Kylin 中需要高精度可以用 bitmap)。

应用了 Hyperloglog 算法的框架:Redis，Apache Kylin

8.7.3 HyperLogLog 王者

​ Set 虽好，如果访问量达到千万级别，一个 Set 就保存了千万个用户的 D，用户多了消耗的内存也太大如果遇到这种情况，最理想的状态是不需要记录用户的编号，只需要有一个人访问就增加一次统计量即可，这样不管你上千万还是上亿的访问都不会增加内存耗费。
​ 这些就是典型的[基数统计] 应用场景，基数统计: 统计一个集合中不重复元素的个数HyperLoglog 的优点在于它所需的内存并不会因为集合的大小而改变，无论集合包含的元素有多少个HyperLogLog 进行计算所需的内存总是固定的，并且是非常少的。
​ 每个 HyperLogLog 最多只需要花费 12KB 内存，在标准误差 0.81%的前提下，就可以计算 2的 64 次方个元素的基数。

9.Redis 的线程问题(选学)

​ 在redis6 版本之前 redis 是单线程模型,这里的单线程不是指 Redis 服务中只有一个线程,而是表示 redis存取数据的核心操作模块由单个线程完成(由一个线程完成)，因为另外还有一些 辅助线程从旁协助，比如LRU 的淘汰过程。
​ 当我们向 redis 保存或者是查询一条数据的时候会经历如下几个过程: ①客户端与 redis 服务端建立连接->②客户端向 redis 服务端发送命令–>③redis 服务端读取(read) 客户端发送的命令->④执行客户端发送的命令—>⑤客户端输出 (write) 执行结果—⑥客户端读取 redis 服务端返回 (write 输出)的结果.

​ 我们来分析③步骤，读取客户端命令有两种策略:
接收客户端命令时一直等，直到接收到完整的命令，然后执行，再将结果返回，直到客户端收到完然后才处理下一个命令， 这叫同步 (服务端和客户端相互等待的模式)。整结果，同步的过程中有很多等待的时间，例如有个客户端网络不好，那等它完整的命令就会更耗时。
​ 客户端的数据包 (命令)来一个才处理一个，将数据追加到缓冲区，处理完了就去立即服务其他客服端，不等待，下一个数据包来了再继续处理。命令的接收过程是穿插的，不连续。一会儿接收这个命令会儿又在接收另一个。这叫做异步，过程中没有额外的空闲等待时间.

​ 很显然异步的效率更高，要实现高并发必须要异步，因为同步有太多时间浪费在等待上了，遇到网络不好的客户端直接就被拖垮。
​ 在 redis 中以上的过程中3步骤都是同一个线程在完成，所以我们说 redis 是单线程的(防止有人咬文嚼字: 这里不是指 Redis 单实例中只有一个线程，而是表示 核心操作 (就是 345 步)模块由单线程完成，当然另外还有一些 辅助线程 从旁协助，比如 LRU 的淘汰过程)
​ redis 主线程的时间消耗主要在两个方面 (主线程就是完成核心操作的线程)：

​ 逻辑计算的消耗(就是处理数据) ，就是对应第四步

​ io读写的消耗，读写数据导致的消耗，对应第 3 和第 5 个步骤当 value 比较大时，瓶颈会先出现在同步 io上，这部分消耗在于两部分:
​ -读取客户端中发送的数据，这个数据包括命令和 value (read 调用)，对应的是第3步骤

​ -将数据回写给到客户端 (write 调用) ，对应的是第5步骤。
​ 于是在 2020 年5月2号时候 redis 推出的6 版本中引入了多线程，redis 6 版本开始将上面的的3和5步骤交个了多个 io 线程线程去完成，④步骤依然由一个线程去完成，据说性能比之前提高了一倍。
