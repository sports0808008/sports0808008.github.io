1. 再次回顾认证概念

   安全框架的核心功能是认证（身份认证）和鉴权（访问控制）。

1.1 session方式的认证

在登录之后将登录信息保存到 session 内置对象,之后在响应中将信息以 cookie 的形式保存到浏览器

◆session.setAttribute( user’,user); 把用户信息存储，其中有返回给客服端的JsessionId

◆在cookie中写了一个JsessionId

◆每次请求都会带上 cookie 信息，cookie 中含有JsessionId，如果JsessionId 在服务端能找到就说明登录了，否则就没有登录

1.2 Token方式的认证

（前后端分离开发模式/分布式的场景）

​ ◆如果一个用户成功登录 (认证成功) 则生成一个对应的 token [token 是通过加密算或者编码 (比如base64编码)法得到的一个字符串,这个 token 要返回一份给浏览器,保存到 cookie 或者/sessionstorage

​ ◆同时把 token 保存一份在 redis (key:token)，也可以保存到 mysgl 数据库(这个步骤叫做 token的持久化)，但是也可以不存，具体要根据需求，要保存的场景: 比如说一个账户同一时间只能一个用户登录 (注意: 这个过程不是单点登录 sso) ，此时保存到 redis 做判断

​ 单点登录(sso）比如：在淘宝登录的在天猫也可以登录

​ 如果 redis 中存在了当前用户的 token 则使用新的 token 覆盖

1.把这个 token 返回给前端 (保存的方式不限，可以保存到 cookie/sessionStorage

2.前端需要把 token 存起来,下次请求的时候都要带上这个 token(一般保存到请求头携带),直接对 token进行校验，校验成功就表示登录了

1.3 两种认证方式的优缺点

Session的方式没有网络开销（优点）不能做单点登录（sso）、 不能做分布式（高可用 高可靠 高可扩） 适用一些后台系统 用户量较小的系统

Token的方式 有网络开销（不足的地方），因为每次发送请求的时候都要带上token会产生一定的网络开销（这都不算什么问题）。

可以做分布式大型的，也可以做单点登录（sso），前后端分离的项目都可以选择这种方式。

2.Token认证的解决方案JWT

Json web token (JWT)，是为了在网络应用环境间传递声明而执行的一种基于JSON****的开放标准，特别适用于分布式站点的单点登录（SSO）场景。JWT的声明一般被用来在身份提供者和服务提供者间传递认证的用户身份信息，以便于从资源服务器获取资源，也可以增加一些额外的其它业务逻辑所必须的声明信息，该token也可直接被用于认证，也可被加密。

2.1 跨域认证问题（了解）

互联网服务离不开用户认证，一般流程是下面这样。

l 用户向服务器发送用户名和密码

l 服务器验证通过后，在当前对话（session）里面保存相关数据，比如用户角色、登录时间等等。

l 服务器向用户返回一个 session_id，写入浏览器的 Cookie。（Cookie不能跨域携带）

l 用户随后的每一次请求，都会通过 Cookie，将 session_id 传回服务器。

l 服务器收到 session_id，找到前期保存的数据进行比对，由此得知用户的身份。

这种模式的问题在于，扩展性（scaling）不好。单机当然没有问题，如果是服务器集群，或者是跨域的服务导向架构，就要求 session 数据共享，每台服务器都能够读取 session。

举例来说，A 网站和 B 网站是同一家公司的关联服务，现在要求，用户只要在其中一个网站登录，再访问另一个网站就会自动登录（这就是单点登录sso），请问怎么实现(？

一种解决方案是 session 数据持久化，写入数据库或别的持久层。各种服务收到请求后，都向持久层请求数据。这种方案的优点是架构清晰，缺点是工程量比较大。另外，持久层万一挂了，就会集群就失败。

另一种方案是服务器索性不保存 session 数据了，所有数据都保存在客户端（浏览器），每次请求都发回服务器。JWT 就是这种方案的一个代表。服务器不存数据，客户端存，服务器解析就行了。

2.2 JWT（json web token） 的原理（了解）

​ 如果要实现remember-me功能，其原理是：登录的时候如果选择了记住我的按钮，则登录成功之后会在后端存一份加密过后的数据，前端的cookie中也保存一份token（也是加密的）。以后的访问中该用户只要访问服务器就会带上token信息，之后要在后端查找对应的加密的token，如果没有则需要登录（可能已经过期了），这种方法一般可以满足百分之70左右的需求了，但是要求严格的系统的可能还不满足需求，因为这种方式是每次都要访问数据库中加密的数据(IO)，造成网络的开销。今天就使用jwt来解决这个问题。

2.3 官网（https://jwt.io/）

Jwt****的特征：登录的时候可以生成一个json格式并且加密的token给到前端，后端不需要存任何东西，以后用户再访问数据库的时候直接带上令牌，服务端获取到token之后进行解密（校验），如果校验成功则可以直接访问资源，否则就不能访问，跳转到登录页面。

JWT 的原理是服务器认证以后，生成一个 JSON 对象，还要将该对象进行加密，发回给用户，就像下面这样。（载荷）

以后，用户与服务端通信的时候，都要发回这个 token信息。服务器完全只靠这个token认定用户身份。为了防止用户篡改数据，服务器在生成这个对象的时候，会加上签名（详见后文）。

服务器就不保存任何 session 数据了，也就是说，服务器变成无状态了，从而比较容易实现扩展。

2.4 JWT 数据的结构(其实要强调的是由哪些部分组成)

实际的 JWT 大概就像下面这样。

它是一个很长的字符串，中间用点（.）分隔成三个部分。注意，JWT 内部是没有换行的，这里只是为了便于展示，将它写成了几行。

JWT 的三个部分依次如下。

Ø Header（头部）

Ø Payload（负载/载荷）

Ø Signature（签名）

写成一行，就是下面的样子。

Header.Payload.Signature

下面依次介绍这三个部分。

2.4.1 Header

Header 部分是一个 JSON 对象，描述 JWT 的元数据，通常是下面的样子。

{

“alg”: “HS256”,

“typ”: “JWT”

}

上面代码中，alg属性表示签名的算法（algorithm），默认是 HMAC SHA256（写成 HS256）；typ属性表示这个令牌（token）的类型（type），JWT 令牌统一写为JWT。

最后，将上面的 JSON 对象使用 Base64URL 算法转成字符串

*了解：base64***算法加密后的数据可能出现+ / =**，

https://www.jianshu.com/p/51589f4cf384（base64加密算法）

https://www.cnblogs.com/marineblog/p/16118851.html****（base64url****算法）

注意：由于 + 和 / 是特殊字符，不能用于url比如上传文件，文件名中有特殊字符。

在Base64Url中，这些特殊字符要替换掉。

  1. + 替换成 -
  2. / 替换成 _
  3. 去掉尾部的 =

2.4.2 Payload

Payload 部分也是一个 JSON 对象，用来存放实际需要传递的数据。JWT 规定了常用的官方字段，供选用。

iss (issuer)：签发人

**exp (expiration time)**：过期时间

sub (subject)：主题

nbf (Not Before)：生效时间

除了官方字段，你还可以在这个部分定义私有字段，下面就是一个例子。

{

“sub”: “1234567890”,

“name”: “John Doe”,

“admin”: true

}

注意，JWT 默认是不加密的，任何人都可以读到，所以不要把敏感信息（密码，手机号等）放在这个部分。这个 JSON 对象也要使用 Base64URL 算法转成字符串。

2.4.3 Signature（保证数据安全性的）

Signature 部分是对前两部分的签名，防止数据被篡改。

首先，需要指定一个密钥（secret）。这个密钥只有服务器才知道，不能泄露给用户。然后，使用 Header 里面指定的签名算法（默认是 HMAC SHA256），按照下面的公式产生签名。

HMACSHA256(

base64UrlEncode(header) + “.” +

base64UrlEncode(payload)+ “.”+

secret)

算出签名以后，把 Header、Payload、Signature 三个部分拼成一个字符串，每个部分之间用”点”（.）分隔，就可以返回给用户。

2.4.4 Base64URL（转码）

前面提到，Header 和 Payload 串型化的算法是 Base64URL。这个算法跟 Base64 算法基本类似，但有一些小的不同。

JWT 作为一个令牌（token），有些场合可能会放到 URL（比如 api.example.com/?token=xxx）。Base64 有三个字符+、/和=，在 URL 里面有特殊含义，所以要被替换掉：=被省略、+替换成-，/替换成_ 。这就是 Base64URL 算法。

2.5 JWT 的使用方式【重点】

​ 客户端收到服务器返回的 JWT（token），可以储存在 Cookie 里面，也可以储存在 localStorage****。

​ 此后，客户端每次与服务器通信，都要带上这个 JWT****。你可以把它放在 Cookie 里面自动发送，但是这样不能跨域，所以更好的做法是放在 HTTP 请求的头信息Authorization字段里面。

Authorization: Bearer jwt

另一种做法是，跨域的时候，JWT 就放在 POST 请求的数据体里面。

2.6 JWT 的几个特点

JWT 默认是不加密，但也是可以加密的。生成原始 Token 以后，可以用密钥再加密一次。

JWT 不加密的情况下，不能将秘密数据写入 JWT。

JWT 不仅可以用于认证，也可以用于交换信息。有效使用 JWT，可以降低服务器查询数据库的次数。

JWT 的最大缺点是，由于服务器不保存 session 状态，因此无法在使用过程中废止某个 token，或者更改 token 的权限。也就是说，一旦 JWT 签发了，在到期之前就会始终有效，除非服务器部署额外的逻辑（JWT的登出问题）。就是因为服务端无状态了

正常情况下 修改了密码后就会跳转到登录页面 ：修改成功后清空浏览器保存的token了

因为服务端不保留token 我用之前的token 还是可以继续访问的

从有状态（后端也会存一个）的变成无状态的了，我们就要把它从无状态再变成有状态了

JWT，对于一些比较重要的权限，使用时应该再次对用户进行认证。

为了减少盗用，JWT 不建议使用 HTTP 协议明码传输，要使用 HTTPS 协议传输。

3.4 JWT签名和验签流程（扩展）

34.1 JWT签名流程

​ (1)创建JWT 的头部和载荷: 在生成WT 之前，需要构建JWT 的头部和载荷。头部包含有关令牌的元数据(指定算法和类型 ，载荷包含有关令牌主题和其他信息的声明。这两部分通常是 JSON 格式的，并目以 Base64 URL 编码的形式编码
​ (2)选择签名算法:JWT 头部通常包含有要使用的签名算法的信息，例如”HMAC SHA256”或”RS256”。这指明了用于生成签名的算法
​ (3)获取密钥: 对于 HMAC 算法，你需要一个密作为输入，这是一个共享的对称密钥，发送方和接收方都必须拥有相同的密钥。对于 RSA 或 ECDSA 等非对称算法，发送方使用自己的私钥来生成签名，接收方使用发送方的公钥来验证签名。
​ (4)生成签名信息:使用指定的签名算法和密钥，对编码后的头部和载荷数据进行签名(加密)。生成的签名是 JWT 的一部分
​ (5)创建JWT: 将编码后的头部、载荷和生成的签名组合在一起，形成一个 JWT

3.4.2 JWT验签流程

​ (1)接收 JWT: 接收方获取 JWT（token令牌）。

​ (2)解码JWT: 接收方首先对JWT 进行 Base64Url 解码以获得头部和载荷信息

​ (3)获取签名算法:接收方使用解码后的 头部信息来确定使用的签名算法是哪一种签名算法，密钥是知道的 (共享密钥)。
​ (4)验证签名:接收方使用相同的签名算法和密钥 对编码后的头部和载荷部分进行签名，然后将生成的签名与JWT 中的签名部分进行比较。如果签名一致，JWT 有效;否则，被视为无效
(5)使用JWT 数据:如果验证成功，接收方可以信任WT 的数据，包括载荷中的声明，以实现认证和授权等功能。

4.3 注意

​ JWT (JavaJSON Web Token) 库在验证JWT 签名时不需要显式指定签名算法，因为它能够自动从JWT 的头部信息中获取签名算法，并使用它进行验证。即使JWT 头部中指定的算法是错误的，JWT 也可以正常进行验签，这是因为JWT 默认会尝试多种算法来验证签名
​ 这种行为是因为JWT 标准允许在头部(Header) 中指定签名算法(alg 字段)，但实际验证时，并不一定要依赖于头部中的指定算法。如果头部中的算法与实际使用的算法不匹配，JWT 库会尝试使用其他常见的算法来验证签名，以确保 WT 的验证不会因此失败。

7.补充过滤器链（了解）

即使大家没有仔细研究过 Spring Security 中认证、授权功能的实现机制，大概也都多多少少听说过 Spring Security 这些功能是通过过滤器来实现的。

是的，没错！Spring Security 中一共提供了 32 个过滤器，其中默认使用的有 15 个。

在一个 Web 项目中，请求流程大概如下图所示：

请求从客户端发起（例如浏览器），然后穿过层层 Filter，最终来到 Servlet 上，被 Servlet 所处理。

那有要问了，Spring Security 中默认的 15 个过滤器就是这样嵌套在 Client 和 Servlet 之间吗？

上图中的 Filter 我们可以称之为 Web Filter，Spring Security 中的 Filter 我们可以称之为 Security Filter，它们之间的关系如下图：

可以看到，Spring Security Filter并不是直接嵌入到 Web Filter 中的，而是通过 FilterChainProxy 来统一管理 Spring Security Filter，FilterChainProxy 本身则通过 Spring 提供的 DelegatingFilterProxy 代理过滤器嵌入到 Web Filter 之中。

DelegatingFilterProxy 在 Spring 中手工整合 Spring Session、Shiro（另外一个安全框架） 等工具时都离不开它，现在用了 Spring Boot，很多事情 Spring Boot 帮我们做了，所以有时候会感觉 DelegatingFilterProxy 的存在感有所降低，实际上它一直都在。
