3.StringRedisTemplate使用

​ 上次课我们知道了RedisTemplate操作redis是将数据转化成字节数组（key和value都会被转换成字节数组）再进行保存的。

​ 如果当你要向redis中存取的数据就是字符串类型的时候（不需想RedisTemplate一样将数据进行序列化和反序列化了），那么你就使用StringRedisTemplate就可以了，

​ 但是如果你的数据是复杂的对象类型（比如自定义的类型的对象），而取出的时候又不想做任何的数据转换，此时可以将对象转换成json格式的数据再进行存储。

​ 如果发现 Redis 中没有保存数据，可能是因为没有正确使用代理类对象来实现业务操作。这里假设您在业务类中创建了一个代理类对象以进行相关操作。

要确保使用代理类对象进行业务操作并将数据保存到 Redis 中，您可以按照以下步骤进行操作：

  1. 创建代理类对象：在业务类中创建一个代理类对象，用于调用相关的方法并处理业务逻辑。

  2. 调用代理类对象方法：在代理类对象中，实现需要进行操作的方法，并确保在适当的步骤将数据保存到 Redis 中。这可以通过在方法中添加 Redis 缓存操作逻辑来实现。比如在方法中先检查 Redis 缓存中是否存在对应的数据，如果存在，则直接返回缓存数据；如果不存在，则从关系型数据库中进行查询，并将结果保存到 Redis 缓存中。

  3. 确保代理类对象之间的相互调用：如果在代理类对象中进行方法调用，确保是使用代理类对象之间的相互调用，而不是直接调用本类中的方法。这样才能保证代理类对象的业务操作被正确执行，并将数据保存到 Redis 中。

通过以上步骤，您可以使用代理类对象来实现业务操作，并确保正确将数据保存到 Redis 缓存中，以便下次查询时可以直接从缓存中获取数据，提高查询效率。

5. 分布式锁的概念

5.1 在传统单体应用单机部署的情况 下存在的问题

​ 为了保证一个方法或属性在高并发情况下的同一时间只能被同一个线程执行，可以使用并发处理相关的功能进行互斥控制。但是，随着业务发展的需要，原单体单机部署的系统被演化成分布式集群系统后，由于分布式系统多线程、多进程并且分布在不同机器上（夸JVM进程），这将使原单机部署情况下的并发控制锁策略失效，原始的互斥控制就不能提供分布式锁的能力。为了解决这个问题就需要一种跨机器的互斥机制来控制共享资源的访问，这就是分布式锁要解决的问题！

5.2 在分布式开发的系统

​ 分布式使用的场景很多，下面我们就选择一个集群环境下的秒杀场景来做说明

​ 接下来我们还是以存秒杀业务场景作为背景进行分布式锁的分析。

      上面秒杀的场景中可能出现一个问题，因为有很多用户参与秒杀，所以可能出现当前用户秒杀时候读取的库存信息是上一个参与秒杀的用户未提交的数据（比如还剩下1件商品，A用户秒杀成功减库存为0，但是还没提交事务的时候B用户查询到库存还是1，继续参与秒杀并且成功，出现了超卖），这样就会导致出现超卖的现象，在之前单服务器部署项目的时候我们可以使用传统的互斥锁解决这一问题，但是现在是分布式项目，所以之前传统的锁无法实现我们的需求了，于是分布式锁就登场了。

​ 要解决这个问题就得使用分布式锁的思想来实现了，分布式锁的实现思路： 第一个用户先到则马上去到redis中去注册锁，注册成功则认为获取了锁，取得当前秒杀的操作权，当库存管理和订单处理业务都执行完毕之后就去redis中去注销（释放锁），后面来访问的用户需要到redis中查看业务是否被其他用户注册了，如果被注册了就等待释放锁。

5.1 总结

​ 我们上面的案例背景是电商中的秒杀业务场景，当然实际的项目中的秒杀会更为复杂，我们这里是简化出来给你们讲解的（也就是整个过程的设计是以你们能理解分布式锁而设计的）
