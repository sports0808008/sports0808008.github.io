1.Nginx的基本概念

分析一下传统的单web服务器存在的问题：

·如果一个项目访问量很高的时候，比如说每秒上万的访问量的时候一个web服务器的负担非常重，此时访问就会变慢（比如果说在实际的应用中的秒杀业务的时候就课程出现高并发访问的情况）甚至导致服务瘫痪宕机。

·如果一个服务器的情况下，宕机的时候的客户是无法访问的。

此时我们使用代理来解决这一问题，使用某种程序算法来代理多个web服务器，就算宕机其中某些服务器，那么用户还可以访问没有宕机的服务器。代理可以将访问分发到多台web服务器，这就是负载均衡的概念，负载均衡的本质就是使用多个web服务器分担压访问压力，负载均衡在很多地方出现了这个概念，比如说数据库的负载均衡、消息中间件的负载均衡等（只要牵涉到集群就会出现负载均衡）。

今天使用Nginx实现web服务器的反向代理。

通过对上图分析我们可以通过反向代理服务器实现负载均衡，这样的代理服务器有很多种，其中最著名和常用的就是Nginx，以上的部署实现了解决高并发访问，但是如果Nginx服务宕机则整个系统就瘫痪了，此时应该解决高可用问题，可以配置多个Nginx服务。

3.反向代理的概念与实现

给银行借钱的例子，你向银行借钱，然后银行借给了你， 但是这个钱大部分是其他用户的存款借给你。

这里的银行扮演着一个反向代理的角色，借钱的人不知道这个钱是谁存的。

反向代理隐藏了真实的服务端，当我们访问www.baidu.com的时候，背后可能有成千上万台服务器为我们服务，但具体是哪一台，你不知道，也不需要知道，你只需要知道反向代理服务器是谁就好了，www.baidu.com直接访问到的是代理服务器。

代理服务器会帮我们把请求转发到提供真实服务的服务器那里去。Nginx就是性能非常好的反向代理服务器，它可以用来做负载均衡。

  1. try_files $uri $uri/ /index.html
  2. try_files：-尝试访问对应的静态资源，如果找不到则访问第二个资源，依次向后查找
  3. $uri ：Nginx地址变量，即为访问的地址
  4. 若访问url为 http://www.xxx.com/update 则 $uri 为 /update
  5. $uri/ ：表示一个目录，请求访问的目录，Nginx try_files可自行判断访问目的的类型 是为文件还是目录
  6. 若访问url为 http://www.xxx.com/user/class/ 则 $uri/ 为 /user/class/

所以以上配置的规则为 当 $uri 和 $uri/ 均不是对应资源时 则返回 /index.html 页面

5.负载均衡策略

负载均衡用于从“upstream”模块定义的后端服务器列表中选取一台服务器接受用户的请求。

5.1 轮询（默认策略）

　　最基本的配置方法，上面的例子就是轮询的方式，它是upstream模块默认的负载均衡默认策略。每个请求会按时间顺序逐一分配到不同的后端服务器。

　有如下参数：fail_timeout 与max_fails结合使用。

max_fails 设置在fail_timeout参数设置的时间内最大失败次数，如果在这个时间内，所有针对该服务器的请求都失败了，那么认为该服务器会被认为是停机了，

fail_time 服务器会被认为停机的时间长度,默认为10s。

backup 标记该服务器为备用服务器。当主服务器停止时，请求会被发送到它这里。

down 标记服务器永久停机了。

　注意：

在轮询中，如果服务器down掉了，会自动剔除该服务器。

缺省配置就是轮询策略（默认）。

此策略适合服务器配置相当，无状态且短平快的服务使用。

5.2 权重（第二种负载均衡策略）

比如你的主机的性能各不相同，有的性能很高，应该让性能高的多处理请求（能者多劳）。

如下图：

此时其中一种解决方案是让某个用户一直访问第一次访问到的web服务器，可以在配置上加上ip _hash，以上的方式是Nginx本身支持一种操作，还有一种操作是将登录的信息保存到redis中，之后在访问Nginx的时候要每次都要到redis中去获取查询（检测）登陆信息判断是否登录。

还有一种方式是使用令牌解析的方式完成（这种更好），这是我们之前讲解jwt的时候分析过。

◆ip_hash配置

此时测试发现我们第一次访问的是第二台服务器，那么后面的访问都是被绑定了只i能访问第二台。

注意：

1、由于ip_hash机制为每个ip绑定了一个固定的web服务器，可能会导致某个服务器接收的请求比较多，某些web服务器接收的请求较少，无法保证web服务器的负载均衡，这是ip_hash的一个缺点，因此不建议使用。

2、Ip_hash不能和backup一起使用在nginx之前版本。

5.4 least_conn（最少连接负载均衡）

　　把请求转发给连接数较少的后端服务器。轮询算法是把请求平均的转发给各个后端，使它们的负载大致相同；但是，有些请求占用的时间很长，会导致其所在的后端负载较高。这种情况下，least_conn这种方式就可以达到更好的负载均衡效果。

upstream appserver{

​ least_conn; #把请求转发给连接数较少的后端服务器

​ server localhost:8080 weight=2; #tomcat 7.0

​ server localhost:8081; #tomcat 8.0

​ server localhost:8082 backup; #tomcat 8.5

​ server localhost:8083 max_fails=3 fail_timeout=20s; #tomcat 9.0

}

注意：

此负载均衡策略适合请求处理时间长短不一造成服务器过载的情况。

5.5 第三方策略（了解）

　　 第三方的负载均衡策略的实现需要安装第三方插件。

①fair

　　按照服务器端的响应时间来分配请求，响应时间短的优先分配。响应时间短表示能力强。

#动态服务器组

upstream appserver{

​ server localhost:8080; #tomcat 7.0

​ server localhost:8081; #tomcat 8.0

​ server localhost:8082; #tomcat 8.5

​ server localhost:8083; #tomcat 9.0

​ fair; #实现响应时间短的优先分配

}

②****url_hash （有点像ip_hash，固定到一台服务器上）

　　按访问url的hash结果来分配请求，使每个url定向到同一个后端服务器，要配合缓存命中来使用。同一个资源多次请求，可能会到达不同的服务器上，导致不必要的多次下载，缓存命中率不高，以及一些资源时间的浪费。而使用url_hash，可以使得同一个url（也就是同一个资源请求）会到达同一台服务器，一旦缓存住了资源，再此收到请求，就可以从缓存中读取。　

upstream appserver{

​ hash $request_uri; #实现每个url定向到同一个后端服务器

​ server localhost:8080; #tomcat 7.0

​ server localhost:8081; #tomcat 8.0

​ server localhost:8082; #tomcat 8.5

​ server localhost:8083; #tomcat 9.0

}

7.proxy_set_header配置（选学）

proxy_set_header其实就是请求头的相关信息的设置，在这里我们了解一下就行了，nginx****代理请求头常用配置项如下。

nginx****代理常用配置项通过以下的案例来理解proxy_set_header设置

proxy_set_header Host &host;

将域名或者ip保存到请求头中,就是nginx做代理时,把原http请求的Header中的Host字段也放到转发的请求里,如果不加这一行的话,nginx转发的请求header里就不会有Host字段。

proxy_set_header X-Real-IP $remote_addr;
只能获取到与服务器本身直连的上层请求ip 写入到http请求头中

proxy_set_header X-Real-IP $proxy_add_x_forwarded_for;
定义客户端IP,如果多层代理都有配置这个值,离backend最近的会覆盖之前的值 所以一般只在连接客户端的代理上写添加这条值,这样backend就能获取到客户端的真实IP

proxy_set_header X-Forwarded-For **$**http_x_forwarded_for;

把客户端IP写入到http请求头中,前提是之前有X-Real-IP $remote_addr定义过客户端IP

proxy_set_header X-Forwarded-For **$**proxy_add_x_forwarded_for;
相当于 $http_x_forwarded_for 和 $remote_addr和结合 获取上层IP和把客户端IP写入请求头中,

8. 缓存的配置（选学）

​ 使用的Nginx代理多个web容器的时候，如果将信息显示给客户，那么需要从各个web服务器将数据传输到Nginx之后再显示给客户，但是有很多数据（静态资源），比如说css、js、图片等信息是一般不会经常改变，这些不会改变的资源在访问的时候每次web服务器都要重新发送给Nginx，这样就会造成资源的浪费，最好的做法是将静态的资源直接缓存到Nginx服务器，之后用户在访问的就直接从Nginx服务器取得这些静态资源，就不从web服务器获取，从而提高了相应的速度，还可以节省流量。

但是这些配置我们只需要掌握核心的地方就行了，你能把这个配置复制过去让缓存生效即可，不需要深入研究每个配置的。

1、建立缓存数据的目录：

2、修改Nginx的配置文件nginx.conf

3、基本配置分析

proxy_cache_path /usr/data/nginx/cache levels=1:2 keys_zone=cache_one:200m inactive=1d max_size=30g; 缓存的基本信息配置
location ~ ..(gif|jpg|png|htm|html|css|js|flv|ico|swf)(.) { proxy_pass http://appserver ; proxy_redirect off; //不转发 proxy_set_header Host $host:$server_port; proxy_cache cache_one; proxy_cache_key $host$uri$is_args$args; proxy_cache_valid 200 302 1h; proxy_cache_valid 301 1d; proxy_cache_valid any 1m; expires 30d; //指定的是这些静态资源在客户端浏览器的缓存时间 } 所有要缓存的数据类型
location ~ /purge(/.*) { allow 127.0.0.1; allow 192.168.139.0/24;【表示139网段的都能访问】 allow all; proxy_cache_purge cache_one $host$1$is_args$args; } allow 192.168.139.0/24;【表示139网段的都能访问】 allow all;【表示所有的都能访问】

上述参数说明如下：

父元素 名称 描述
http proxy_cache_path 指定缓存区的根路径
levels 缓存目录级最高三层，每层1-2个字符表示。如1:1:2三层。
keys_zone 缓存块名称及内存块大小。如cache_item:500m表示声明一个名为cache_item大小为500m。超出大小 后最早的数据将被清除。
max_size 缓存区硬盘的最大值。超出闲置数据将被清除
inactive 最长闲置时间 如10d 如果一个数据被闲置10天则将被清除
location proxy_cache 指定缓存区，对应keys_zone中设定的值
proxy_cache_key 通过参数拼装参数key如：$host$uri$is_args$args则会以缓存数据的全部路径的md5值做为key
proxy_cache_valid 对不同的状态码设置缓存有效期
