动态代理概念

​ 不管是动态代理还是静态代理，其作用为某些方法做增强（不会改变这些方法的结构），

​ 被增强的方法叫做切点，动态代理本身就是面向切面编程实现的底层的思想（AOP）。

​ 在之前我们讲解设计模式的时候已经分析了静态代理，但是静态代理会存在一个弊端，就是一个代理类只能作为一个接口的代理（只能为一种接口服务），如果要实现一个代理类可以作为任意接口的实现类代理则需要使用到动态代理来完成，在学习动态代理之前我们先回顾静态代理。

JDK 动态代理的不足

​ 在 JDK 中使用动态代理，必须有类的接口（被代代理的对象必须有自己父接口）。因为生成的代理类$Proxy0 需要实现这个接口，这样我们生成的代理类对象，才能转化为代理目标的接口对象，然后根据接口中的方法，调用处理器中 invoke 方法。

Cglib 动态代理

​ 为了弥补 JDK 动态代理的不足（被代理的对象必须要有自己的接口），第三方组织封装一套工具包，cglib 的工具包，这套包不基于接口（不在需要被代理的类必须有接口），基于父子继承关系（生成的代理对象的父类是被代理的类型），通过重写的形式扩展方法，但是这个子类也是自动生成的（也不会持久化）。

​ 早期，Cglib 动态代理，性能比 JDK 的动态代理高一些。**JDK 进行一些列优化，目前 Spring 的 AOP 默认使用的动态代理就是 JDK 提供的，也支持 Cglib(但是需要进行配置)**。

CGLIB
通过动态生成一个需要被代理类的子类（即被代理类作为父类），该子类重写被代理类的所有不是
final
修饰的方法，并在子类中采用方法拦截的技术拦截父类所有的方法调用，进而织入横切逻辑（在执行业务方法之前取得连接）。

​ 在底层实现上，CGLIB 使用字节码处理框架
ASM，该框架用于转换字节码并生成新的类。但是不鼓励直接使用 ASM，因为它要求对于
JVM 的内部结构包括 class 文件的格式和 JVM 指令都很熟悉，若一旦出现错误，会导致
JVM 崩溃级别的异常。

​ 虽然说 CGLIB 与 JDK 动态代理相比，具有更好的优势，当并不是说使用 CGLIB
会更好，这里先总结下 CGLIB 和 JDK 动态代理之间的区别：

**|-**JDK 动态代理只能对接口进行代理，不能对普通的类进行代理，这是因为 JDK
动态代理生成的代理类，其父类是  Proxy，同时还需要实现被代理类的接口。

**|-**CGLIB
能够代理接口和普通的类，但是被代理的类的反方以及代理类本身不能被  final  修饰，且接口中的方法不能使用  final  修饰。

**|-**JDK 动态代理使用 Java 反射技术进行操作，在生成类上更高效。

**|-**CGLIB 使用 ASM
框架直接对字节码进行修改，在某些情况下，类的方法执行会比较高效。

AOP 的概念

​ 通过之前动态代理的学习，我们知道如果要操作数据库中的数据，那么需要先取得连接，操作完毕之后需要关闭连接，除了这些基本的操作之外还可能在操作数据之前要进行权限验证、还可能操作完毕之后需要记录操作日志（操作数据的时间、用户的基本信息、操作的电脑的 ip 地址等等）、还需要进行事务处理，以上这些操作基本上每个操作数据的方法都可能要辅助执行，此时就出现了重复得代码。为了解决这样的重复的代码，提出了一种开发思想：面向切面的编程，叫做AOP 编程（Aspect
Oriented
Propramming），就是将那些辅助的操作（打开连接、关闭连接、权限验证…）放到一个专门的类中实现，这个类可以横切需要操作数据的类的方法。
